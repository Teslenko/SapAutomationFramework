# Roadmap: SAP .NET Automation Framework

Этот документ описывает поэтапный план развития собственного `.NET 8 / C# / NUnit` фреймворка
для автоматизации SAP GUI, который должен стать заменой существующим UiPath-тестам.

## Бизнес-цели

- Показать руководству жизнеспособность переноса SAP-тестов с UiPath на собственный .NET framework.
- Снизить зависимость от лицензий UiPath.
- Получить обычный код на C# с Git, Code Review и CI/CD вместо закрытых xaml-воркфлоу.
- Предоставить читаемый, декларативный API в стиле Playwright.
- Обеспечить полуавтоматическую миграцию существующих UiPath-тестов.
- Упростить поиск локаторов SAP GUI без ручного анализа дерева интерфейса.

## Целевой поток вызовов

```text
NUnit Tests
    ↓
Page Objects / Business Flows
    ↓
ISapDriver
    ↓
SapGuiScriptingDriver
    ↓
SAP GUI Scripting COM API
    ↓
SAP GUI for Windows
```

## Текущее состояние (as-is)

- Решение `SapAutomation.sln` с двумя проектами: `SapAutomation.Core` и `SapAutomation.Tests` (NUnit), таргет `net8.0`.
- В `SapAutomation.Core` заложен каркас Clean Architecture:
  - `Abstractions/` — `ISapAutomationEngine`, `ISapConnection`, `ISapSession`, `ISapGuiElement`;
  - `Models/` — `ConnectionOptions`, `SessionInfo`;
  - `Exceptions/` — `SapAutomationException`.
- Зависимостей на SAP COM пока нет (согласно `CLAUDE.md`), бизнес-сценарии SAP не реализованы.
- Реальной реализации драйвера, Page Object API, поиска локаторов и миграции из UiPath ещё нет — это предмет данного roadmap.

## Этапы

### Phase 0 — Фундамент (готово)

- Чистая архитектура: `Core` / `Tests`, разделение на `Abstractions` / `Models` / `Exceptions`.
- NUnit как тестовый фреймворк, `dotnet build` / `dotnet test` проходят.
- Явный отказ от SAP COM-зависимостей и бизнес-логики на этом этапе.

### Phase 1 — Слой драйвера (`ISapDriver` → `SapGuiScriptingDriver`)

- Ввести `ISapDriver` — верхнеуровневую абстракцию над сессией SAP GUI (запуск, подключение, поиск элементов, ожидания).
- Реализовать `SapGuiScriptingDriver` поверх официального `SAP GUI Scripting API / COM` (`SapROTWr.dll` / `GuiApplication`).
- Реализовать конкретные обёртки для `ISapConnection`, `ISapSession`, `ISapGuiElement` на основе COM-объектов SAP GUI.
- Добавить обработку типовых ошибок SAP GUI Scripting (недоступность скриптинга, обрыв соединения) через `SapAutomationException`.
- Изоляция COM-зависимости в отдельном слое/проекте, чтобы `Core` оставался тестируемым без установленного SAP GUI.

### Phase 2 — Page Objects / Business Flow API в стиле Playwright

- Спроектировать декларативный, читаемый API поверх `ISapDriver` (fluent-обёртки, аналог `page.locator(...)`).
- Базовые классы для Page Object / Business Flow, инкапсулирующие транзакции и экраны SAP.
- Стратегия локаторов (id, path, свойства элемента) с единым API поиска и явными ожиданиями.
- Примеры простых, не бизнес-специфичных сценариев для демонстрации API (login-экран, стандартные диалоги).

### Phase 3 — Инструмент поиска локаторов SAP GUI

- Утилита/режим "инспектора", которая подключается к активной сессии SAP GUI и строит дерево элементов с их id/свойствами.
- Экспорт дерева локаторов в читаемый формат (JSON/markdown) для переиспользования в Page Objects.
- Опционально: минимальный recorder, фиксирующий клики/ввод пользователя как черновик локаторов (без записи полноценных скриптов).

### Phase 4 — Полуавтоматическая миграция из UiPath

- Анализ типовых конструкций существующих UiPath-проектов (селекторы, sequences, arguments).
- Конвертер/маппер, переводящий UiPath-селекторы SAP GUI в локаторы нового фреймворка.
- Руководство по миграции: соответствие UiPath-активностей (`Click`, `Type Into`, `Get Text`, ...) вызовам нового API.
- Пилотный перенос одного-двух существующих UiPath-тестов как proof of concept.

### Phase 5 — Тестовая инфраструктура, отчётность, CI/CD

- Интеграция отчётности (Allure, согласно стеку в `README.md`) для NUnit-тестов.
- Настройка CI-пайплайна: `dotnet build` + `dotnet test` на каждый PR.
- Правила Code Review для новых Page Objects / Business Flows.
- Базовые guidelines по структуре тестов (Arrange/Act/Assert, изоляция окружения SAP).

### Phase 6 — Пилотный перенос и презентация руководству

- Перенос небольшого, но показательного набора реальных SAP-тестов с UiPath на новый framework.
- Сбор метрик: время миграции, стабильность, читаемость кода, экономия на лицензиях UiPath.
- Презентация результатов руководству как обоснование для дальнейшего масштабирования миграции.

## Ограничения и принципы (сквозные для всех фаз)

- Сохранять `net8.0` и `NUnit` на всех этапах.
- Не добавлять SAP COM-зависимости и бизнес-специфичные сценарии до Phase 1 включительно.
- Изменения в архитектуре предварительно проговаривать (см. `CLAUDE.md`), не переписывать существующий код без необходимости.
- Каждая фаза должна давать самостоятельно демонстрируемый результат (build/test проходят, есть пример использования).
