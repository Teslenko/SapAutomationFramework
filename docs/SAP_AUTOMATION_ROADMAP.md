# Roadmap: SAP .NET Automation Framework

Документ описывает поэтапный план разработки собственного `.NET 8 / C# / NUnit`
фреймворка для автоматизации SAP GUI и оценку его пригодности как замены части
существующих UiPath-тестов у заказчика.

Документ намеренно не содержит календарных сроков и оценок экономии в процентах —
на момент написания нет ни доступа к тестовой системе SAP заказчика с включённым
GUI Scripting, ни доступа к существующим UiPath-проектам, поэтому любые числовые
прогнозы были бы необоснованными.

## Целевой поток вызовов

```text
NUnit Tests
    ↓
Page Objects / Business Flows
    ↓
ISapDriver
    ↓
SapGuiScriptingDriver
    ↓
SAP GUI Scripting COM API
    ↓
SAP GUI for Windows
```

## Таблица этапов

| Phase | Название | Приоритет | Сложность | Краткое содержание |
|---|---|---|---|---|
| Phase 0 | Доступ и готовность окружения | Must | S | Получить доступ к тестовой SAP-системе с включённым GUI Scripting и к образцам UiPath-проектов; подтвердить, что скриптинг не заблокирован политикой ИБ |
| Phase 1 | Core Driver Layer | Must | M | `ISapDriver` и `SapGuiScriptingDriver` поверх SAP GUI Scripting COM API; обёртки `ISapConnection` / `ISapSession` / `ISapGuiElement` |
| Phase 2 | Page Object / Business Flow API | Must | M | Декларативный API в стиле Playwright поверх `ISapDriver`; базовые классы Page Object, стратегия локаторов, явные ожидания |
| Phase 3 | Инструмент поиска локаторов | Should | M | Утилита, подключающаяся к активной сессии SAP GUI и строящая дерево элементов с их id/свойствами, без ручного анализа UI |
| Phase 4 | Анализ UiPath и помощники миграции | Should | L | Разбор структуры существующих UiPath-проектов (селекторы, sequences), полуавтоматические помощники переноса селекторов в локаторы фреймворка; ручная проверка результата обязательна |
| Phase 5 | Тестовая инфраструктура и CI/CD | Must | M | Отчётность (Allure) для NUnit, пайплайн `dotnet build` + `dotnet test` на каждый PR, правила Code Review |
| Phase 6 | Pilot PoC | Must | L | Перенос ограниченного набора не бизнес-критичных сценариев с UiPath на новый фреймворк как доказательство концепции |
| Phase 7 | Оценка результатов PoC | Should | S | Разбор ограничений и проблем, обнаруженных в PoC; решение о целесообразности продолжения |
| Phase 8 | Масштабирование | Could | XL | Расширение переноса на дополнительные сценарии/команды при положительном решении по Phase 7 |

Приоритет: **Must** — необходимо для работоспособного фреймворка; **Should** — важно для практического использования, но не блокирует PoC; **Could** — рассматривается только при положительном решении по итогам PoC.

Сложность (S/M/L/XL) — относительная, без привязки к календарным срокам.

## Детали этапов

### Phase 0 — Доступ и готовность окружения (Must, S)

На момент написания документа:

- нет доступа к тестовой/sandbox-системе SAP заказчика;
- не подтверждено, что SAP GUI Scripting включён и разрешён политикой ИБ заказчика;
- нет доступа к существующим UiPath-проектам/тестам заказчика (селекторы, структура workflow).

Это блокирует начало Phase 1 и Phase 4 соответственно. Пока доступ не получен,
работа ограничена кодом, не зависящим от SAP COM (см. текущее состояние
`SapAutomation.Core`: абстракции, модели, исключения, NUnit-тесты без реального
подключения к SAP).

### Phase 1 — Core Driver Layer (Must, M)

- `ISapDriver` — абстракция над сессией SAP GUI (запуск, подключение, поиск элементов, ожидания).
- `SapGuiScriptingDriver` — реализация поверх `SAP GUI Scripting API / COM`.
- Конкретные реализации `ISapConnection`, `ISapSession`, `ISapGuiElement`.
- Обработка типовых ошибок SAP GUI Scripting через `SapAutomationException`.
- COM-зависимость изолируется в отдельном слое, чтобы `Core` оставался тестируемым без установленного SAP GUI.

### Phase 2 — Page Object / Business Flow API (Must, M)

- Fluent-API поверх `ISapDriver`, читаемый в стиле Playwright.
- Базовые классы Page Object / Business Flow для экранов и транзакций SAP.
- Единая стратегия локаторов (id, path, свойства элемента) с явными ожиданиями.
- Демонстрационные (не бизнес-специфичные) сценарии для проверки API.

### Phase 3 — Инструмент поиска локаторов (Should, M)

- Инспектор, подключающийся к активной сессии SAP GUI и строящий дерево элементов с их id/свойствами.
- Экспорт дерева локаторов в читаемый формат (JSON/Markdown) для переиспользования в Page Objects.
- Возможное расширение: минимальная запись действий пользователя как черновика локаторов (без генерации полноценных скриптов).

### Phase 4 — Анализ UiPath и помощники миграции (Should, L)

Важно: **полностью автоматическая миграция не планируется**. Цель этапа —
инструменты и правила, ускоряющие ручной перенос, а не заменяющие его.

- Разбор типовых конструкций существующих UiPath-проектов (селекторы, sequences, arguments).
- Помощник, переводящий UiPath-селекторы SAP GUI в черновые локаторы нового фреймворка — результат требует ручной проверки и правки.
- Руководство по соответствию UiPath-активностей (`Click`, `Type Into`, `Get Text` и т.п.) вызовам нового API.
- Перенос одного-двух существующих тестов вручную, с использованием помощников, как проверка подхода.

### Phase 5 — Тестовая инфраструктура и CI/CD (Must, M)

- Интеграция отчётности Allure для NUnit-тестов.
- CI-пайплайн: `dotnet build` + `dotnet test` на каждый PR.
- Правила Code Review для новых Page Objects / Business Flows.
- Базовые guidelines по структуре тестов и изоляции окружения SAP.

### Phase 6 — Pilot PoC (Must, L)

- Перенос небольшого набора не бизнес-критичных, воспроизводимых сценариев с UiPath на новый фреймворк.
- Сценарии выполняются через NUnit против реальной SAP-системы с включённым GUI Scripting.
- Фиксация реально возникших проблем и ограничений (без прогнозов на будущее).

### Phase 7 — Оценка результатов PoC (Should, S)

- Разбор того, что сработало и что нет по итогам Phase 6.
- Качественное (не в процентах) сравнение с эквивалентным UiPath-тестом: читаемость, стабильность, объём ручной работы.
- Решение о целесообразности Phase 8.

### Phase 8 — Масштабирование (Could, XL)

- Расширение переноса на дополнительные сценарии/транзакции при положительном решении по Phase 7.
- Объём и состав определяются по итогам PoC, а не заранее.

| Risk | Impact | Probability | Mitigation | Owner | Status |

| Нет доступа к тестовой SAP-системе с GUI Scripting | High | High | Запросить доступ к sandbox/dev-системе до начала Phase 1 | Заказчик | Open |
| Нет доступа к существующим UiPath-проектам | High | High | Запросить экспорт/доступ к образцу проектов до начала Phase 4 | Заказчик | Open |
| SAP GUI Scripting запрещён или ограничен политикой ИБ | High | Medium | Подтвердить у SAP Basis / ИБ заказчика включённость и разрешённость скриптинга заранее | Заказчик / SAP Basis | Open |
| UiPath-селекторы слишком разнородны для типового преобразования | Medium | Medium | Рассматривать помощник миграции как черновик, требующий ручной проверки, а не автоконвертер | Команда фреймворка | Open |

| Организационное сопротивление отказу от UiPath | Medium | Medium | Представлять результаты PoC прозрачно, без завышенных обещаний | Спонсор проекта | Open |

| Нестабильность/недоступность тестового SAP-окружения | Medium | Medium | Согласовать выделенное тестовое окно с командой Basis заказчика | Заказчик | Open |

## Definition of Done для первого PoC

PoC (Phase 6) считается завершённым, если выполнено всё нижеперечисленное:

- `ISapDriver` / `SapGuiScriptingDriver` подключается к реальной сессии SAP GUI через COM и стабильно открывает/закрывает сессию.
- Минимум один не бизнес-критичный, воспроизводимый сценарий (например: вход в систему → переход в транзакцию → чтение значения поля) автоматизирован через Page Object API от начала до конца.
- Сценарий выполняется через NUnit и проходит локально на реальной системе с включённым SAP GUI Scripting.
- Локаторы, использованные в PoC, получены через инструмент поиска локаторов (Phase 3) либо задокументированы вручную, если инструмент ещё не готов — без "угадывания" id на глаз.
- `dotnet build` и `dotnet test` проходят зелёным в CI.
- Зафиксировано короткое качественное сравнение с эквивалентным UiPath-тестом (без придуманных цифр).
- Задокументирован список реально обнаруженных ограничений и блокеров.

## Open Questions

- Когда и как будет предоставлен доступ к тестовой SAP-системе с включённым GUI Scripting?
- Какие конкретно UiPath-проекты/тесты будут доступны для анализа и пилотного переноса?
- Есть ли ограничения информационной безопасности на использование SAP GUI Scripting API в тестовой и/или продуктивной среде?
- Кто со стороны заказчика владеет решением о масштабировании после PoC (Phase 7 → Phase 8)?
- Какой минимальный набор сценариев считается достаточным для PoC — по объёму и по типу транзакций?
- Как будет организован доступ к CI/CD-инфраструктуре заказчика, или используется отдельная инфраструктура команды фреймворка?
- Кто выполняет Code Review со стороны заказчика, если он требуется?

Примечание: список открытых вопросов составлен на основе текущего состояния
проекта (отсутствие доступа к SAP GUI и UiPath-проектам заказчика) и требует
подтверждения/дополнения по мере уточнения требований.

## Ограничения и принципы (сквозные для всех фаз)

- Сохранять `net8.0` и `NUnit` на всех этапах.
- Не добавлять SAP COM-зависимости и бизнес-специфичные сценарии до завершения Phase 0.
- Не обещать полностью автоматическую миграцию из UiPath — только полуавтоматические помощники с обязательной ручной проверкой.
- Не указывать числовые оценки экономии или календарные сроки без фактических данных.
- Изменения в архитектуре предварительно проговаривать, не переписывать существующий код без необходимости.
