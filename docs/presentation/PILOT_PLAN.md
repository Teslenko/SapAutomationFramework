# Pilot Plan

Этот план описывает, что уже реально реализовано в коде, что нужно от банка
для следующего шага и как будет выглядеть первая проверка на настоящем SAP.
Всё, что перечислено в разделе "Сделано", подтверждено зелёными
`dotnet build` / `dotnet test` — но **исключительно на Mock Driver**
(`SapAutomation.Infrastructure.Mock`) и, для API-модуля, на публичном
эхо-сервисе httpbin.org. Ни одного прогона против реального SAP GUI ещё не
было.

## Что уже сделано

- **`SapAutomation.Core`** — абстракции фреймворка: `ISapConnection`,
  `ISapSession`, `ISapGuiElement`, `ISapTextBox`, `ISapButton`,
  `ISapStatusBar`, `ISapLogger`; модели `ConnectionOptions`/`SessionInfo`;
  единое исключение `SapAutomationException`.
- **`SapAutomation.Infrastructure.Mock`** — полная in-memory реализация всех
  абстракций выше (без SAP COM). Позволяет разрабатывать и проверять
  архитектуру фреймворка без доступа к реальному SAP.
- **`SapAutomation.Infrastructure.Logging`** — структурированное логирование
  действий (`Action | ElementId | Duration | Result`); пароль в
  `ConnectionOptions` маскируется при логировании и никогда не пишется как
  есть.
- **Page Object / Business Flow (демо)** — `Fb60Page` (Page Object для
  транзакции FB60) и `InvoiceFlow.PostVendorInvoice` (Business Flow) —
  читаемый API в стиле Playwright, отработан на Mock Driver.
- **`SapAutomation.Api`** — независимый REST/OData-клиент (`IApiClient`,
  `HttpApiClient`), провайдеры авторизации (Bearer/Basic/API-Key).
  Используется как для тестирования API напрямую, так и как прекондишен для
  UI-сценариев (создание/удаление тестовых данных). Продемонстрировано на
  публичном httpbin.org — реального SAP API у проекта пока нет.
- **Отчётность по тестам** — автоматическая генерация `TestResults/report.md`
  после каждого прогона NUnit (тест, бизнес-шаг, статус, длительность,
  сообщение об ошибке при падении).
- **10 зелёных NUnit-тестов** — все либо на Mock Driver, либо против
  httpbin.org; ни один не затрагивает реальный SAP.

## Спроектировано, но не реализовано

- **Waits** (`WaitUntilExists`/`WaitUntilVisible`/`WaitUntilText` и т.п.) и
  **fluent Assertions** (`ShouldContain`/`ShouldBe`/`ShouldNotContain` с
  отдельным исключением) — обсуждались и был согласован план реализации, но
  код по нему ещё не написан.
- **`ISapDriver` / `SapGuiScriptingDriver`** — реальный слой поверх SAP GUI
  Scripting COM API. Это Phase 1 по `docs/SAP_AUTOMATION_ROADMAP.md` и
  прямо зависит от доступа к тестовой SAP-системе.
- Инструмент поиска локаторов и помощники миграции из UiPath (Phase 3–4 по
  roadmap) — не начаты.

## Что нужно от банка для следующего шага

- Доступ к тестовой/sandbox SAP-системе с включённым SAP GUI Scripting
  (см. `RISK_MATRIX.md`, риск №11 — на сегодня trial ожидает одобрения).
- Подтверждение от ИБ/SAP Basis, что SAP GUI Scripting разрешён политикой
  безопасности (в том числе при запуске не с локальной машины разработчика).
- Ответы на Open Questions из `docs/SAP_AUTOMATION_ROADMAP.md` — в первую
  очередь: кто владеет решением о масштабировании после пилота и какой
  минимальный набор сценариев считается достаточным для PoC.
- Доступ/экспорт образцов существующих UiPath-проектов — не блокирует
  первый SAP smoke-тест, но нужен для последующего анализа миграции
  (Phase 4).

## Что будет продемонстрировано после получения доступа

Первый реальный SAP smoke-тест — соответствует Phase 1 roadmap ("Core Driver
Layer"), а не Phase 3 (та посвящена отдельному инструменту поиска
локаторов). Объём:

- `SapGuiScriptingDriver` подключается к реальной сессии SAP GUI через COM и
  стабильно открывает/закрывает сессию.
- Один не бизнес-критичный сценарий (например: вход в систему → переход в
  транзакцию → чтение значения поля) выполняется через существующий Page
  Object API на реальном SAP GUI вместо Mock Driver.
- Зафиксированы реально обнаруженные ограничения и отличия от поведения на
  Mock Driver — без прогнозов "что, вероятно, будет работать".

## Критерии успеха пилота (Definition of Done)

Совпадают с Definition of Done для PoC из `docs/SAP_AUTOMATION_ROADMAP.md`:

- `SapGuiScriptingDriver` подключается к реальной сессии SAP GUI через COM и
  стабильно открывает/закрывает сессию.
- Минимум один не бизнес-критичный, воспроизводимый сценарий автоматизирован
  через Page Object API от начала до конца на реальном SAP.
- Сценарий выполняется через NUnit и проходит локально на системе с
  включённым SAP GUI Scripting.
- Локаторы, использованные в PoC, получены через инструмент поиска
  локаторов либо задокументированы вручную — без угадывания id.
- `dotnet build` и `dotnet test` проходят зелёным в CI.
- Зафиксировано короткое качественное сравнение с эквивалентным UiPath-тестом
  (без придуманных цифр экономии).
- Задокументирован список реально обнаруженных ограничений и блокеров.
