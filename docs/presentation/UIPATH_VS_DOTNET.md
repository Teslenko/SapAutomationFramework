# UiPath vs собственный .NET-фреймворк

Сравнение структурное и качественное — без точных цифр стоимости/экономии и без
прогноза сроков. Часть критериев (в первую очередь скорость выполнения) пока не
проверена ни для одного из вариантов на реальной системе SAP заказчика: весь
опыт с собственным фреймворком на сегодня получен на in-memory Mock Driver
(`SapAutomation.Infrastructure.Mock`), не на реальном SAP GUI.

| Критерий | UiPath | Собственный .NET-фреймворк | Комментарий |
|---|---|---|---|
| Лицензии/стоимость | Требует лицензии (Studio/Orchestrator/Robot — конкретная модель зависит от контракта) | Не требует лицензий поверх .NET SDK и NUnit (оба бесплатны и open-source) | Отсутствие лицензии не означает "бесплатно" — команда всё равно тратит время на разработку и поддержку фреймворка |
| Контроль версий и Code Review | Проекты в формате `.xaml`; diff/review читаются плохо, слияние конфликтов затруднено | Обычный C# в Git: полноценный `git diff`, PR, Code Review как для любого кода | Явное преимущество .NET-подхода — совпадает с целью проекта (см. roadmap) |
| Читаемость тестов | Визуальный workflow — читается как диаграмма, но плохо виден в виде текста/diff | Читаемый C#/NUnit в стиле Playwright (Page Object + Business Flow, например `InvoiceFlow.PostVendorInvoice`) | Подтверждено демо-сценарием на Mock Driver |
| Скорость выполнения | Не измерялась в рамках этого проекта | Не измерялась на реальном SAP; на Mock Driver действия занимают единицы миллисекунд, но это не показатель — Mock не имитирует задержки реального SAP GUI | Честно: сравнение скорости появится только после Phase 1 (реальный SAP-драйвер) и smoke-теста |
| Порог входа для новых инженеров | Ниже для low-code/RPA-специалистов без опыта программирования | Требует знания C#/.NET; ниже порог для .NET-разработчиков, выше — для RPA-инженеров без опыта кода | См. риск "недостаток C#-компетенций у QA" в `RISK_MATRIX.md` |
| Гибкость/расширяемость | Ограничена набором активностей и маркетплейсом коннекторов | Полный контроль над кодом и архитектурой (Core/Mock/Api-слои уже разделены), но всё нужно писать самим | Возможность добавлять произвольную логику без обхода UI-редактора — на данный момент только в архитектуре, без реального SAP-слоя |
| Экосистема готовых коннекторов | Существенно шире — сотни готовых activities/интеграций из UiPath Marketplace | Практически с нуля: пока есть только базовый HTTP/REST-клиент (`SapAutomation.Api`) и слой абстракций для SAP GUI (без реализации над реальным COM) | Здесь UiPath объективно выигрывает — это не скрывается |
| Windows-зависимость | Да (SAP GUI Scripting/COM — технология Windows) | Да, по той же причине | Одинаковое ограничение для обоих — следствие SAP GUI Scripting API, а не архитектурного решения какой-либо из сторон |
| Зрелость решения | Годы промышленной эксплуатации, большое сообщество | Разработка ведётся несколько недель; ни разу не запускался против реального SAP GUI | Существенный разрыв в зрелости — фреймворк ещё не прошёл ни одного реального прогона |

## Итог

Собственный фреймворк выигрывает там, где была цель проекта: контроль версий,
Code Review, читаемость кода. UiPath выигрывает в зрелости, экосистеме
коннекторов и пороге входа для не-программирующих RPA-инженеров. Скорость
выполнения и реальная стабильность на SAP GUI пока не проверены ни для одного
варианта в рамках этого сравнения — вывод по ним возможен только после
получения доступа к SAP и первого реального smoke-теста (см. `PILOT_PLAN.md`).
